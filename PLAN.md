# Hayku Agent — 企業軟體建構自主代理

## 願景

建立一個高度自主的 Agent，能夠持續記憶、自行擴展工具，並為企業建構所需的軟體系統（Web app、內部工具、資料管線、API 服務），整合一致性的企業資訊，提升效率。

## 設計原則

- 多模型架構 — 不鎖定單一 LLM
- 框架不限定 — 根據需求選擇最適技術
- 高度自主 — Agent 自行決定整合什麼系統、使用什麼技術
- 工具自生成 — Agent 判斷任務需要時自行建立新工具
- 安全優先 — 自建工具必須經過驗證閘門

### 核心模組說明

| 模組 | 職責 |
|------|------|
| **Planner** | 拆解任務、決定策略、判斷是否需要新工具 |
| **Executor** | 執行工具、管理工作流程、處理錯誤 |
| **Memory** | 短期記憶（對話上下文）+ 長期記憶（向量/結構化 DB） |
| **Tool Manager** | 工具註冊、自動生成、沙箱驗證、版本控制 |
| **Model Router** | 根據任務複雜度/成本/速度選擇不同 LLM |

## 自建工具安全驗證機制

### 1. 沙箱執行（Sandbox）
- 新工具在隔離環境中執行（容器、WASM、受限 process）
- 限制檔案系統存取範圍、網路存取、系統呼叫
- 即使工具有 bug，爆炸半徑被控制

### 2. 靜態分析 + 自我審查
- Agent 生成工具後，用另一個 LLM session 做 code review
- 檢查：刪除操作、外部網路呼叫、敏感資料存取
- 規則引擎（禁止 `eval`、禁止直接 SQL 拼接等）

### 3. 權限分級制度

| 等級 | 範圍 | 策略 |
|------|------|------|
| Level 0 | 純計算（字串處理、資料轉換） | 自動放行 |
| Level 1 | 讀取本地檔案 / DB | 記錄 log |
| Level 2 | 寫入檔案 / DB、呼叫外部 API | 需人工審批或二次確認 |
| Level 3 | 系統層操作（安裝套件、修改設定） | 必須人工審批 |

### 4. 測試驗證閘門
- 建立工具的同時必須產生測試案例
- 測試通過才能註冊到工具庫
- 定期回歸測試確保持續正確

### 5. 版本控制 + 回滾
- 每個工具有版本，所有變更進 git
- 出問題可立即回滾到上一個可用版本

### 6. 審計日誌（Audit Trail）
- 記錄：觸發者、建立原因、輸入/輸出、影響範圍
- 滿足企業合規要求

## 記憶系統設計

### 短期記憶
- 當前對話上下文
- 進行中的工作狀態與任務佇列

### 長期記憶
- **向量 DB** — 語意搜尋，用於模糊匹配過去的經驗和知識
- **結構化 DB** — 精確查詢，用於工具 registry、企業資料模型、設定
- **檔案系統** — 生成的程式碼、文件、工具原始碼

### 記憶索引策略
- 每次任務開始時，根據任務描述做語意搜尋，召回相關記憶
- 關鍵決策和學習到的經驗主動寫入長期記憶
- 定期清理過時或低價值記憶
